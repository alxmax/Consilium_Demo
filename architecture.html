What is Consilium?

A deliberation pattern that evaluates code changes through three parallel voices with separate roles (Generator / Control / Conservator), aggregates votes via a veto-enabled scheme, and produces a decision plus a canonical report.

The core idea: never rely on a single perspective for a risky change. Each voice is prompt-constrained from doing the others' job — separation of powers implemented through context stripping (Sequential — Blind) or independent sub-agents (Parallel / Dialectic).

Used for large refactors, DB migrations, security work, or any change where you want a structured "second opinion" before merge. Output lands as canonical JSON in runs/ and as a line in FEEDBACK.md for future calibration.

Constitution — governs every deliberation
1. Think before coding — surface tradeoffs, don't assume.
2. Simplicity first — minimum code that solves the problem.
3. Surgical changes — touch only what the goal demands.
4. Goal-driven — success_criterion + verification are mandatory.

The 3 voices

Generator

Creative
Produces 3-5 candidate approaches via divergent thinking. Always includes do_nothing as a baseline.
  • Curious, not cautious — risk is someone else's job
  • Quantity before quality
  • No self-censorship
Not allowed: to filter for risk or feasibility.

Control

Analytical
Technically checks each candidate: types, logic, tests, style. Marks valid: true/false.
  • Pedantic, not pessimistic
  • Concrete over abstract
  • Verifies, doesn't speculate
Not allowed: to reject for risk — pass through if correct.

Conservator

Skeptical
Scores risk for each valid candidate: diff_size, scope_drift, regression_risk, reversibility.
  • Reversibility over cleverness
  • Blast radius matters
  • May recommend do_nothing
Not allowed: to re-validate correctness — trust Control.

Aggregation — 4 voting schemes

majority Mean of votes; tiebreak on minimum variance.
weighted Weighted mean with weights ∈ [0.2, 0.49], sum=1.0.
conservative_override DEFAULT Weighted mean with safety = 1 − conservator; Conservator holds unilateral veto. VETO at risk_score > 0.8
risk_adjusted_utility utility = mean(gen, ctrl, 1−cons) with a sigmoidal penalty centered at risk=0.5. No hard veto — a candidate with risk 0.7 is penalized ~21%, not disqualified. Use when multiple candidates cluster around the threshold.

Numeric example — same candidate set, 4 different winners

Three candidates with identical utility (gen, ctrl) but increasing conservator (risk). Each scheme aggregates differently — the first two treat conservator as utility (historical bug); the last two flip it into safety. Numbers computed with equal weights = ⅓.
candidate generator control conservator (risk)
A — safe0.800.900.30
B — medium0.900.800.60
C — risky0.850.850.85
scheme score(A) score(B) score(C) winner
majority 0.6670.7670.850 C
weighted 0.6670.7670.850 C
conservative_override 0.8000.700VETO A
risk_adjusted_utility 0.7050.1880.018 A
Why the default is conservative_override: majority and weighted treat the conservator score as utility (higher = better), so they pick C — the riskier candidate. Only schemes that flip conservator into safety (1 − risk) reach the intuitive verdict. risk_adjusted_utility yields the same result without a cliff at the threshold — useful when several candidates cluster around 0.8.
Separation of powers: each voice is prompt-constrained from doing the others' job. Constitution overrides any voice in case of conflict. If all candidates are vetoed → the aggregator attaches a retry_suggested block with the minimum threshold under which the lowest-risk candidate would have survived (capped at 0.85). The decision (re-roll with a stricter constraint, accept chosen: null, or ask the user) falls to the main agent.

Step 5b + 6 — confidence gate decision tree

How the outcome written to FEEDBACK.md is decided after aggregation. scripts/confidence.py derives the score; scripts/log_feedback.py gates it at 0.7.
Outcomes are OK, OVR, PEND, BAD. The tree covers the automatic flow at the end of step 6 — additionally, at the confidence < 0.7 prompt the user may choose to defer the rating (equivalent to running log_feedback.py without --outcome → PEND, retroactively closed via stale_pendings), and --outcome BAD is set manually later to mark a decision as failed.

Feedback loop — deliberation → calibration → deliberation cycle

The loop, in words: at the end of each deliberation, build_report.py assembles the canonical report, validate_report.py verifies Principle #4 (shape + telemetry.mode + deliberation_log), and log_feedback.py appends a line to FEEDBACK.md according to the tree above. At the start of the next run, priors.py reads runs/ + FEEDBACK.md and computes the soft priors the agent injects into prompts — the skill thus self-calibrates its risk/override intuition based on actual history, not just defaults.

Trias — deep dive

How information flows between voices in Trias, a concrete output example, and when it is (and isn't) the right choice.

1. How information flows

2 isolation levels: between personalities via chinese wall (level 1 — separate sub-agents); between voices within each personality via context strip (level 2 — sequential, as in Sequential — blind). Single-direction flow — the orchestrator collects at the end, never during the deliberation.
cross-level flow (orchestrator ↔ personalities)
intra-personality flow (voice → voice, as in Sequential — blind)
chinese wall — personality isolation (level 1)

3. Cost & failure modes

✓ USE TRIAS WHEN
  • Irreversible schema/DB migration (ALTER TABLE on millions of rows)
  • Security audit — auth paths, crypto, RCE potential, secrets handling
  • Large refactor — touches ≥5 files or modifies core interfaces
  • 2+ plausible architectural approaches, no clear winner
  • Cost-of-wrong-decision >> cost-of-running (3 sub-agents)
✗ DO NOT USE WHEN
  • Diff < 20 lines / 1 file — scope_gate.py will skip anyway
  • Fast iteration — Trias is ~30-60s, ~7× cost vs Sequential
  • Strict conservatism required — Trias aggregate is −18% Conservator
  • Obvious bugfix, single approach — Sequential blind suffices
  • Throwaway code / experiments / ad-hoc scripts
Failure modes & recovery
Failure Trigger Recovery
1-1-1 fragmentation 3 personalities, 3 distinct chosen results → no majority Escalate to the user; or re-run Generator with added constraints ("avoid X, Y, Z")
0-0-0 total veto All personalities vetoed all candidates (risk > 0.8 across the board) retry_suggested with relaxed_threshold; strong signal that the change is risky/bad
Sub-agent timeout A voice doesn't respond within budget (rare — Sonnet 4.6 is stable) Emits null for the affected personality; the others vote 2-0 or 1-0
Inconsistent veto Conservator scoring differs across personalities for the same candidate (variance > 0.3) Log warning in telemetry.notes; non-blocking but a signal of instability
Steward abstain Steward = null, others have chosen (2-0 vote) OK auto at confidence 0.75, but Steward's veto reasoning is flagged separately — a stronger signal than Pioneer/Architect abstention
Max 3 sub-agents active simultaneously in standard modes (1 + 2 pattern); up to 9 in Trias (3 personalities × 3 voices). The prompts/*.md prompts are self-contained — they can be handed directly to a sub-agent with no adjustments.

Calibration loop

Between deliberations, the Reviewer logs real outcomes in FEEDBACK.html. A small calibration job reads the log, computes per-voice stats, and adjusts prompts and thresholds before the next deliberation runs. This is not part of any per-run flow — it is the maintenance loop that keeps the system honest.

Reviewer marks past run as OK / BAD / OVR
FEEDBACK.html append-only outcome log
feedback.py stats & priors across runs/
Prompt tuning weights · veto threshold · prompts/*.md
↻ Tuned prompts and thresholds apply to the next deliberation — the loop closes without code changes.

Benchmark — modes × tasks

A local harness that runs each Consilium mode against three tasks using claude -p — Claude Code's headless, non-interactive subprocess mode. Each call is fire-and-forget: the runner spawns claude -p "<task prompt>", collects output, then grades it automatically via verify.py. No browser, no user input, no interactive prompts — the model must solve the task in a single headless session.

Isolation: each run is sandboxed to its own workspace/<mode>/<task>/ directory (the model's cwd). The scoring/ tree — rubrics, expected answers, test files — lives outside prompts/ and is never in the model's context. A post-run behavior audit scans the session transcript and voids any cell where scoring files were accessed during the run.

Per-run parameters: Sonnet 4.6 · --effort high · $1.5 budget cap ($3.0 for multi-agent modes) · 15-minute wall-clock cap.

Task 1 — Circuit Breaker

Code · C++17 Hard 
Implement a circuit-breaker state machine in C++17.
Three states: CLOSED → OPEN → HALF_OPEN → (CLOSED | OPEN).

Required interface (header-only, solution.hpp):

  CircuitBreaker(int failure_threshold,
                 int recovery_timeout_sec,
                 int success_threshold);

  void record_success();
  void record_failure();
  bool is_open() const;   // true ONLY in OPEN state
  State state() const;    // CLOSED / OPEN / HALF_OPEN (lazy evaluation)

Key rules:
- CLOSED: consecutive failures trip OPEN; any success resets the streak.
- OPEN: transitions to HALF_OPEN lazily on query after recovery_timeout_sec.
        record_success/failure are ignored while inside the open window.
- HALF_OPEN: is_open() returns false (probe calls allowed).
             record_success() re-closes after success_threshold hits.
             record_failure() immediately re-opens (full timeout resets).

Deliverables: solution.hpp + tests_self.cpp (exit 0 = pass).
