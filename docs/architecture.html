Consilium

Pattern de deliberare multi-perspectivă pentru orice modificare de cod — 3 voci · 8 moduri · 7 senatori
Începe cu Overview pentru ce e + cum curge. Apoi Modes pentru când folosești fiecare flux. Tab-ul curent se reține pe sesiune.

Ce este Consilium?

Un pattern de deliberare care evaluează schimbări de cod prin trei voci independente (Generator / Control / Conservator), agregă voturile printr-o schemă cu veto, și produce o decizie + un report canonic în runs/.

Folosit pentru refactor, migrări DB, security, sau orice schimbare unde vrei un "second opinion" structurat înainte de merge. Output-ul ajunge ca JSON canonic în runs/ și ca linie în FEEDBACK.html pentru calibrare viitoare.

Constitution — guvernează fiecare deliberare
1. Think before coding — expune tradeoff-uri, nu presupune.
2. Simplicity first — minimum de cod care rezolvă problema.
3. Surgical changes — atinge doar ce cere goal-ul.
4. Goal-driven — success_criterion + verification obligatorii.

STEP 0

Bootstrap

Citește contractele celor 3 voci. Rulează priors.py — întoarce soft priors din FEEDBACK.html + runs/. Blochează dacă stale_pendings > 0 sau missing_feedback_runs > 3.

priors.py → stale_pendings, pend_pressure, weighted_bad_rate
STEP 1

Gather & Goal

Citește schimbarea propusă. Formulează success_criterion testabil. Clarity gate: dacă există 2+ interpretări → Stop, listează-le.

input: diff + context → success_criterion
STEP 1.5

Scope Gate

Auto-skip dacă diff e trivial (max_files=1, max_lines=15) sau blocklist hit. Gate eșuează deschis (no repo → skip=false).

scope_gate.py → should_skip: true|false
STEP 2

Conservator RUNS FIRST

Evaluează risc. Output setează tokens_budget pentru Steps 3-4. irreversibility_flag:true blochează pipeline — cere confirmare user.

conservator.md → {risk_score, tokens_budget, meta_recommendation}
STEP 3

Generator

Produce 3-5 candidați inclusiv do_nothing. Respectă tokens_budget.generator. Adversarial condițional.

generator.md → {candidates[], preferred, fallback_scenario}
STEP 4

Control

Per-candidat: types → logică → tests → style. tests_to_write obligatoriu pentru valid:true. Primește full output Conservator + Generator.

control.md → {verdicts[], glossary, disagreements}
STEP 5 → 5d

Aggregate + Confidence

aggregator.py --scheme conservative_override → chosen. Apoi confidence.py (5b), meta_critic.py (5c). Dacă conf < 0.7 → retry_context.py (5d) un singur retry.

→ {chosen_approach, confidence, deliberation_quality.flags}
STEP 6

Report

build_report.py | validate_report.py. Persistă în runs/. Loghează în FEEDBACK.html via log_feedback.py. Gate: exit 0=OK, exit 1=field gol, exit 2=JSON malformat.

runs/YYYY-MM-DD_HHMM_<label>.json → FEEDBACK.html

Aggregation — 3 scheme active (+ 1 deprecated)

majority Media voturilor; tiebreak pe varianță minimă.
weighted (deprecated) Medie ponderată cu weights ∈ [0.2, 0.49]. Scoasă din CLI — tratează conservator ca utility. Funcția rămâne importabilă cu DeprecationWarning.
conservative_override (default) Medie ponderată cu safety = 1 − conservator; Conservator are veto unilateral. VETO la risk_score > 0.8
risk_adjusted_utility utility = mean(gen, ctrl, 1−cons) cu penalty sigmoidal centrat la risc=0.5. Fără veto rigid.

Three-layer architecture (RUND2)

Consilium are 3 layere distincte. Deliberation rulează pe fiecare întrebare user. Aggregation decide ce vede user-ul. Senate intervine doar când e modificat skill-ul însuși.

Deliberation
Conservator → Generator → Control (sequential, default RUND2)
Rulează pe fiecare întrebare user. Auto cross-check parallel se declanșează doar la magnitude: critical ȘI reversibility: irreversible. Silent audit la fiecare 20 runs. Parallel nu mai e selectabil de user.
Aggregation
aggregate_rund2() cu 8-component veto cascade → 7 routing outcomes
Sintetizează output-urile vocilor. Decide BLOCK / REWORK / ADAPT_SHORT / ADAPT_EXTENDED / ESCALATE / AGGREGATE. Veto budget: 5 activări per lună.
Senate
7 senatori (Wittgenstein, Aurelius, Confucius, Socrate, Musk, Dimon, Napoleon)
On-demand audit pentru schimbări la skill-ul însuși. Niciun trigger automat. Verdictul e advisory. Vezi tab-ul Reference → Senate.

Voices

Trei voci core (Generator / Control / Conservator) + 3 personality lenses pentru Trias + Skeptic focal + Domain lens experimental. Această pagină este sursa canonică unică pentru descrierile vocilor — alte taburi referențiază vocile doar după nume.

Generator

CREATIVE

Curious, not cautious. Rulează al doilea în pipeline, după Conservator.

  • Cantitate înainte de calitate. Cinci candidate mediocre bat un singur "perfect".
  • Zero auto-cenzură. Dacă o abordare pare ciudată, o listezi oricum.
  • Include trivialul. do_nothing și revert sunt mereu pe masă.
  • Respectă tokens_budget. Conservatorul a calibrat cât merită această deliberare.

Rol: Produce 3–5 candidate meaningful-diferite: do_nothing obligatoriu; adversarial_* când clarity gate ridică 2+ readings; unconventional_* dacă adversarial nu acoperă deja acel rol.

Nu are voie: să filtreze pentru risc sau fezabilitate — acela e Conservatorul. Să editorializeze în rationale despre pericole — asta e downstream. Să depășească semnificativ tokens_budget.generator.
Contract: prompts/voices/generator.md

Control

ANALITIC

Pedant, nu pesimist. Rulează al treilea, după Conservator și Generator.

  • Concret peste abstract. „Va arunca pe input gol" bate „ar putea avea edge cases".
  • Verifică, nu specula. Dacă nu poți verifica un tip fără să citești fișierul, citești fișierul.
  • Standard uniform. Aceleași criterii pentru fiecare candidate, fără excepție.
  • 4 întrebări obligatorii înainte de validarea per-candidate: glossary, hidden assumptions, disagreements, constraints.

Rol: Per-candidate verdict: types → logic → tests → style → goal-fit. tests_to_write obligatoriu pentru orice valid: true (excepție do_nothing). glossary_fail: true → soft veto.

Nu are voie: să respingă un candidate pentru risc — dacă e corect, valid: true. Să marcheze invalid din motive estetice — acelea merg la Conservator. Să producă glossary cu definiții abstracte neancorate în deliberarea curentă.
Contract: prompts/voices/control.md

Conservator

RISK

Semnal de risc, nu decizie. Rulează primul — output-ul lui calibrează efortul Generator + Control.

  • Reversibilitate peste deșteptăciune. O schimbare plictisitoare dar reversibilă bate una ingenioasă dar permanentă.
  • Status quo bias check. Distinge „ireversibil pentru real" de „ireversibil pentru că schimbarea e incomodă".
  • Self-scaling. Decizii triviale-reversibile = output de 2 propoziții. Critice-ireversibile = analiză completă.
  • Poate recomanda do_nothing. Uneori calea cea mai sigură e inacțiunea.

Rol: Scor net_concern ∈ [0,1] din 4 componente (diff_size, scope_drift, regression_risk, reversibility). Floor rule: dacă reversibility_score > 0.7 → net_concern ≥ reversibility_score. Setează tokens_budget pentru Generator și Control.

Nu are voie: să re-valideze corectitudinea — trust Control (valid: true înseamnă valid). Să confunde „nu-mi place abordarea" cu „e riscantă". Să seteze orice candidate la net_concern: 0.5 — aia nu e judecată, e un shrug.
Contract: prompts/voices/conservator.md

Skeptic — focal challenger

Challenger focal pe un singur candidate ales. Folosit în parallel_skeptic, dialectic_skeptic, și flag-ul composabil skeptic_on_chosen. Nu e un adversar general — operează după ce pipeline-ul a ales un câștigător.

  • Input: Primește doar chosen + success_criterion + verification + context codebase. Nu vede ceilalți candidați, nu vede scorurile Conservatorului, nu vede verdictele Control.
  • Regulă centrală — concret sau tăcut. O obiecție vagă e mai rea decât nicio obiecție. can_object: true necesită ≥2 concerns concrete SAU ≥1 scenariu citat. Sub acest prag → can_object: false, fără fabricație.
  • meta_scope_mismatch: Dacă chosen e corect dar întreaga deliberare a fost over-aplicată pe o problemă trivială (rezolvabilă de un om în <10s), marchează failure_mode: "meta_scope_mismatch" + addressable: "unaddressable".
  • Addressability: in_place (fix mic) · requires_redesign (alt candidate) · unaddressable (escaladare).

Schema output: { can_object, objection: { concrete_concerns[], quoted_scenario, failure_mode, addressable }, notes } · Prompt: prompts/voices/skeptic.md · Advisory by default; override via --skeptic-can-override.

How Consilium remembers

Consilium nu deliberează în izolare — ține minte decizii trecute pe 3 layere cu lifecycle diferit. Loop-ul complet e deliberate → outcome → priors → next deliberation.

scripts/priors.py citește long memory la Step 0 și injectează soft signals în deliberarea curentă. Outcome-ul scris la Step 6 nu e final — mark_outcome.py permite revizie retroactivă cu marker [confirmed] (cântăresc 2× față de subiective).

Git workflow + self-improvement loop

1. Branch nou de la main înainte de a edita

Naming: feat/<slug> sau fix/<slug>. Slug kebab-case. Doar aceste 2 prefixe.

2. Un singur commit per branch (amend strategy)

Primul git commit, apoi git commit --amend --no-edit la fiecare modificare ulterioară din aceeași sesiune.

3. Înainte de push, întreb explicit

"Totul ok sau mai vrei schimbări?" Dacă cere modificări → amend + reia întrebarea.

4. Push o singură dată → git checkout main

După push, nu mai amend + force-push fără cerere explicită. Schimbări noi = branch nou.

5. PR-ul îl face utilizatorul manual

Nu rulez gh pr create. La final, doar raportez branch-ul push-uit.

6. Mesajele de commit = Conventional Commits

Format: feat(scope): ... / fix(scope): .... Omit Co-Authored-By: Claude ... pe Consilium.

7. Excepție typo / fix 1-linie

Poate merge direct pe main dacă utilizatorul cere explicit. Restul intră în workflow-ul de mai sus.

Când editezi skill-ul însuși, rulează /consilium pe propria schimbare.
→
Pentru schimbări la prompts/voices/*.md, prompts/senators/*.md sau arhitectură → preferă modul senate (7 senatori, mai puternic decât single-context).
→
Salvează run-ul în runs/ și loghează în FEEDBACK.html prin log_feedback.py.
