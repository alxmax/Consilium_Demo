Consilium

Multi-perspective deliberation pattern for any code change — 3 voices · sequential, dialectic, trias, skeptic
Start with Overview for what it is + how it flows. Then Modes for when to use each flow. The current tab is remembered per session.

What is Consilium?

A deliberation pattern that evaluates code changes through three independent voices (Generator / Control / Conservator), aggregates votes through a veto scheme, and produces a decision + a canonical report in runs/.

Used for refactoring, DB migrations, security, or any change where you want a structured "second opinion" before merge. The output lands as canonical JSON in runs/ and as a row in FEEDBACK.html for future calibration.

Why 3 voices, not one?

A single LLM evaluating its own proposal has predictable blind spots: it generates ideas and validates them from the same perspective. Three voices with disjoint mandates (creative · analytical · risk) cross-validate before aggregation.

Generator

creative · runs second
in: diff + tokens_budget · out: candidates[] + preferred

Control

analytical · runs third
in: candidates + full output · out: verdicts[] + glossary

Conservator

risk · runs first
in: diff + context · out: risk_score + tokens_budget

Full details: Voices tab.

Constitution — governs every deliberation
1. Think before coding — surface tradeoffs, don't assume.
2. Simplicity first — minimum code that solves the problem.
3. Surgical changes — touch only what the goal requires.
4. Goal-driven — success_criterion + verification mandatory.

Bootstrap

STEP 1

Gather & Goal

Reads the proposed change. Formulates a testable success_criterion. Clarity gate: if 2+ interpretations exist → Stop, list them.

input: diff + context → success_criterion
STEP 1.5

Scope Gate

Auto-skip if diff is trivial (max_files=1, max_lines=15) or blocklist hit. Gate fails open (no repo → skip=false).

STEP 2

Conservator RUNS FIRST

Evaluates risk. Output sets tokens_budget for Steps 3-4. irreversibility_flag:true blocks pipeline — asks for user confirmation.

Generator

Produces 3-5 candidates including do_nothing. Respects tokens_budget.generator. Conditional adversarial.

generator.md → {candidates[], preferred, fallback_scenario}
STEP 4

Control

Per-candidate: types → logic → tests → style. tests_to_write mandatory for valid:true. Receives full Conservator + Generator output.

control.md → {verdicts[], glossary, disagreements}

Report

Two-layer architecture

Consilium has 2 distinct layers. Deliberation runs on every user question. Aggregation decides what the user sees.

Deliberation
Conservator → Generator → Control (sequential, default)
Runs on every user question. Auto cross-check parallel triggers only at magnitude: critical AND reversibility: irreversible. Silent audit every 20 runs. Parallel is no longer user-selectable.
Aggregation
aggregate_rund2() with 8-component veto cascade → 7 routing outcomes
Synthesizes voice outputs. Decides BLOCK / REWORK / ADAPT_SHORT / ADAPT_EXTENDED / ESCALATE / AGGREGATE. Veto budget: 5 activations per month.

Step 1.5 — scope gate decision tree

When to run full deliberation vs. skip directly to report. scripts/scope_gate.py, defaults: max_files=1, max_lines=15.

SKIP (skipped_report)

  • diff ≤ 1 file
  • diff ≤ 15 lines
  • no blocklist hit
  • e.g.: typo, rename, comment fix

DELIBERATE (full pipeline)

  • multi-file OR >15 lines
  • blocklist hit (auth, crypto, migrations)
  • fail-open: probe error → deliberate
  • e.g.: refactor, feature, migration
Override: CONSILIUM_FORCE_FULL=1 forces deliberate. Defaults in DEFAULT_CONFIG; overridden via scope_gate.json or --config PATH. Non-diff tasks (audit, planning): scope_gate is no-op.

Overview Glossary

aggregator
The script that transforms the 3 voice votes into a single chosen_approach. 4 schemes, default conservative_override. scripts/aggregator.py.
veto cascade
Set of 8 triggers derived from voice outputs that decide deliberation routing (BLOCK/REWORK/ADAPT/ESCALATE/AGGREGATE) before aggregation itself.
conservative_override
Default aggregation scheme: weighted average with safety = 1 − conservator, plus unilateral veto for Conservator at risk_score > 0.8.
scope_gate

confidence gate
Step 5b: if confidence < 0.7 after aggregation, triggers single-pass retry with enriched context (Step 5d).
two-layer architecture
Deliberation (3 voices per question) + Aggregation (synthesis output). Distinct layers, different scopes.

What exactly is a "sub-agent"?

It's not a new process or a new model — it's a new API invocation with its own context window and its own prompt. The orchestrator (Claude main) receives back only the final JSON declared in the voice's system prompt. No shared state.

Workflow — 4 steps

  1. Step 1 — Conservator reads diff + context, returns risk_score + tokens_budget + irreversibility_flag. If irreversibility_flag=true → pipeline blocked, asks for user consent.
  2. Step 2 — Generator receives diff + tokens_budget (from step 1), produces 3–5 candidates[] + preferred. Must include do_nothing.
  3. Step 3 — Control receives candidates[] + full_output, validates per-candidate: {id, valid, issues, tests_to_write}. strip_context.py removes the Generator's rationale.
  4. Step 4 — Aggregator runs conservative_override: weighted average with Conservator veto at risk_score > 0.8. Returns chosen_approach + confidence + alternatives.

Workflow — 5 steps

Modes Glossary

chinese wall
Informational isolation between sub-agents — no sub-agent sees another's deliberation. Comes from separate dispatch, not prompt sanitization.
strip_context
Sequential utility — projects the previous voice's output to minimum (id+summary+sketch) before passing it to the next voice. scripts/strip_context.py.
skeptic_on_chosen
Composable flag over any mode — adds 1 sub-agent that challenges chosen post-hoc. Advisory default; override opt-in via --skeptic-can-override.
lens
Additional prompt prepended over the core voice (Pioneer/Architect/Steward) — biases perception without changing the voice's role. Used only in Trias.
magnitude / reversibility
Two dimensions of a change extracted by Conservator: magnitude (small/medium/critical) and reversibility (reversible/expensive/irreversible). The critical+irreversible combination triggers auto cross-check.
scope_gate
Step 1.5: auto-skip for trivial diffs (see Overview glossary).
silent audit
Every 20 runs, Parallel runs automatically alongside Sequential, without affecting output. If systematic divergence appears → frequency increases to 1/5.

The 3 voices — order + what each receives

Generator

CREATIVE

Curious, not cautious. Runs second in the pipeline, after Conservator.

  • Quantity before quality. Five mediocre candidates beat one "perfect" one.
  • Zero self-censorship. If an approach seems strange, list it anyway.
  • Include the trivial. do_nothing and revert are always on the table.
  • Respect tokens_budget. The Conservator has calibrated how much this deliberation is worth.

Role: Produce 3–5 meaningfully-different candidates: do_nothing mandatory; adversarial_* when clarity gate raises 2+ readings; unconventional_* if adversarial doesn't already cover that role.

Not allowed: to filter for risk or feasibility — that's the Conservator's job. To editorialize in rationale about dangers — that's downstream. To significantly exceed tokens_budget.generator.
Contract: prompts/voices/generator.md

Control

ANALITIC

Pedantic, not pessimistic. Runs third, after Conservator and Generator.

  • Concrete over abstract. "Will throw on empty input" beats "might have edge cases".
  • Verify, don't speculate. If you can't verify a type without reading the file, read the file.
  • Uniform standard. Same criteria for every candidate, without exception.
  • 4 mandatory questions before per-candidate validation: glossary, hidden assumptions, disagreements, constraints.

Role: Per-candidate verdict: types → logic → tests → style → goal-fit. tests_to_write mandatory for any valid: true (exception: do_nothing). glossary_fail: true → soft veto.

Not allowed: to reject a candidate for risk — if it's correct, valid: true. To mark invalid for aesthetic reasons — those go to Conservator. To produce a glossary with abstract definitions unanchored in the current deliberation.
Contract: prompts/voices/control.md

Conservator

RISK

Risk signal, not decision. Runs first — its output calibrates the effort of Generator + Control.

  • Reversibility over cleverness. A boring but reversible change beats an ingenious but permanent one.
  • Status quo bias check. Distinguish "truly irreversible" from "irreversible because the change is uncomfortable".
  • Self-scaling. Trivial-reversible decisions = 2-sentence output. Critical-irreversible = full analysis.
  • Can recommend do_nothing. Sometimes the safest path is inaction.

Role: Score net_concern ∈ [0,1] from 4 components (diff_size, scope_drift, regression_risk, reversibility). Floor rule: if reversibility_score > 0.7 → net_concern ≥ reversibility_score. Sets tokens_budget for Generator and Control.

Not allowed: to re-validate correctness — trust Control (valid: true means valid). To confuse "I don't like the approach" with "it's risky". To set any candidate to net_concern: 0.5 — that's not judgment, it's a shrug.

Sequential vs. Trias — how the same voices are dispatched

Sequential — 1 context, 3 roluri

Trias — 9 sub-agents, 3 lenses × 3 voices

3 personality lenses — Pioneer / Architect / Steward

Used exclusively in Trias mode (and trias_split). Each lens is prepended to the core voices of the respective personality — biases perception, doesn't change the voice's role. The orchestrator dispatches 3 × 3 = 9 sub-agents; each personality produces a chosen via weighted aggregation, then voted on democratically.

Skeptic — focal challenger

Focal challenger on a single chosen candidate. Activated via the composable flag skeptic_on_chosen over any base mode (Sequential / Parallel / Dialectic / Trias). Not a general adversary — operates after the pipeline has chosen a winner.

  • Input: Receives only chosen + success_criterion + verification + codebase context. Does not see the other candidates, does not see Conservator scores, does not see Control verdicts.
  • Core rule — concrete or silent. A vague objection is worse than no objection. can_object: true requires ≥2 concrete concerns OR ≥1 cited scenario. Below this threshold → can_object: false, no fabrication.
  • meta_scope_mismatch: If chosen is correct but the entire deliberation was over-applied on a trivial problem (solvable by a human in <10s), mark failure_mode: "meta_scope_mismatch" + addressable: "unaddressable".
  • Addressability: in_place (small fix) · requires_redesign (alternate candidate) · unaddressable (escalation).

Schema output: { can_object, objection: { concrete_concerns[], quoted_scenario, failure_mode, addressable }, notes } · Prompt: prompts/voices/skeptic.md · Advisory by default; override via --skeptic-can-override.

What Skeptic sees / does NOT see

SEES

  • chosen candidate
  • success_criterion
  • verification steps
  • codebase context (Read, Grep)

DOES NOT SEE

  • other candidates
  • Conservator scores
  • Control verdicts
  • the deliberation itself

Informational isolation = Skeptic cannot fabricate objections based on what others have already said; it must find concrete concerns on its own or stay silent (can_object: false).

How Consilium remembers

Consilium doesn't deliberate in isolation — it remembers past decisions across 3 layers with different lifecycles. The full loop is deliberate → outcome → priors → next deliberation.

scripts/priors.py reads long memory at Step 0 and injects soft signals into the current deliberation. The outcome written at Step 6 is not final — mark_outcome.py allows retroactive revision with the [confirmed] marker (weighted 2× vs. subjective ones).

Two-pass: Pass 1 = parallel (isolated). Pass 2 = each voice sees the others' outputs and revises or maintains. Cost: 6 sub-agents (~6× Sequential). Implemented in scripts/dialectic_merge.py.

Why Trias ≈ 9× Sequential — cost anatomy

Cost does not come from deliberation complexity, but from the number of context restarts — each sub-agent re-receives system prompt + diff + context as duplicated input.

Sequential — 1 context, 3 roles

1× input (shared context, stripped between roles)
3 sequential outputs (cons → gen → ctrl)
Cost total: ~1×

Trias — 9 sub-agents, 3 lenses × 3 voices

9× input (each sub-agent full context restart)
9 parallel outputs + democratic vote
Cost total: ~9×

Input tokens dominate cost (system prompt + diff + context = hundreds of thousands of tokens per restart). Output is small by comparison — that is why the multiplier is nearly linear with the number of sub-agents.

When the cost is worth it — cost-of-mistake × cost-of-running

Benchmark methodology — in brief

All modes receive the same --effort flag in benchmark — models decide internally how much to deliberate. We do not tune parameters per-mode. Benchmark problems are designed to challenge models; they are not replaced with simpler variants for cost reduction. See experiments/ for concrete runs (P3 car wash, benchmark wrapper).

Efficiency Glossary

tokens/OK
Total tokens consumed divided by the number of confirmed OK outcomes in the same run set. Key cost-efficiency metric.
baseline
Sequential = 1×. All other costs are reported relative to this.
multiplier vs Sequential
Cost ratio of the mode vs Sequential — e.g. Trias = 9×.
effort flag
--effort low|medium|high in CLI — controls effort intensity per sub-agent (max tokens, retry tolerance). Uniform in benchmark.
OK / PEND / BAD / OVR
The 4 possible outcomes for a run: OK (success), PEND (pending review), BAD (failed in prod), OVR (user override). [confirmed] in note = weights 2× in weighted_bad_rate.
cost-of-mistake
The assumed cost of a wrong decision — irreversibility, blast radius, downtime, lost trust. Used in decision-making for cost-justification (see quadrant).
cost-of-running
The actual token + latency cost of deliberation. Directly proportional to the number of sub-agents.
