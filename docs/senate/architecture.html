§1 — Unde stă Senatul în consilium

3 layere distincte. Senatul e layer-ul de governance — nu intervine pe întrebări regulate, ci doar când skill-ul însuși urmează să fie modificat.

L1 Deliberation layer per-question deliberation
Generator→ Control→ Conservator modes: parallel / dialectic / trias / skeptic*
L2 Aggregation layer vote → verdict → report
aggregator.py→ confidence.py→ build_report.py→ validate_report.py output: runs/<file>.json
L3 Senate layer (governance) on-demand · audit skill changes
7 senators parallel→ senate_synth.py→ verdict GO/MODIFY/STOP/UNREACHABLE output: runs/senate/<file>.json
Reguli izolare: L3 nu se invocă pe întrebări user; L1+L2 nu se invocă pe modificări la skill.

§2 — Cei 7 senatori și lensele lor

Fiecare e o perspectivă cognitivă distinctă. Scope-ul declarat în ## Limite din fiecare prompt; NU se suprapun.

Wittgenstein
operational semantics
„E definit operațional? Cum verifici că s-a respectat?"
→ vague_terms_found[]
Aurelius
reversibility × magnitude
„E proporțional cu stake-ul? Reversibil?"
→ reversibility · magnitude · quadrant
Confucius
hierarchy & precedent
„Cine are autoritate aici? Avem precedente?"
→ hierarchy_check · precedent_search[]
Socrate
hidden assumptions
„Ce presupui fără să declari? Dacă-i falsă?"
→ hidden_assumptions[].load_bearing
Musk
aggressive deletion
„Ce poți șterge fără să pierzi funcția?"
→ components_attacked[].vote
Dimon
stress test & counterparty
„Ce dacă eșuează? Cine plătește?"
→ stress_scenarios[].failure_mode
Napoleon
cost & terrain
„Câți tokens? În ce stare e operatorul?"
→ cost_estimate · terrain_check

Toți emit vote (GO/MODIFY/STOP) și modify_request (string). Ortogonalitatea e validată prin secțiunea ## Limite din fiecare prompt — fiecare declară explicit ce nu face.

§4 — Logica verdictului

Quorum 5/7. Cazul UNREACHABLE (introdus post-self-validation) surfacează când restul de senatori nu pot atinge matematic quorum-ul.

verdict GO
GO ≥ 5/7
Procedezi cu implementarea.
verdict STOP
STOP ≥ 5/7
Revizuie propunerea sau override explicit.
verdict MODIFY
orice altă combinație (default)
Aplică modify_requests și (opțional) re-rulează.
verdict UNREACHABLE
voters_present < 5
Verdict structurally biased. Orchestrator escaladează la user.

                      │  (sau mai puține dacă   │
                      │   senatorii sunt absent)│

                   ┌─────────────────────────────────┐
                   │  voters_present = #(prezenți)   │
                   └────────────────┬────────────────┘
                                    │
                ┌───────────────────┼───────────────────┐
                ▼                                       ▼
        voters_present < 5                       voters_present ≥ 5
                │                                       │

De ce default-ul e MODIFY: Senatul e advisory. GO și STOP sunt convergențe puternice (≥5/7); orice altă combinație = semnal de dezbatere = MODIFY.

Principii de izolare

  1. Zero modificări la voci core sau scripts deliberative. Predicat testabil: git diff <base>..HEAD -- prompts/{generator,control,conservator,skeptic}*.md scripts/{aggregator,confidence,...}.py returnează empty.
  2. Storage separat. runs/senate/ e gitignored (runs/senate/*.json) — istoric local parallel cu runs/*.json.
  3. Cleanup-safe. Dacă ștergi prompts/senators/ + scripts/senate_synth.py + runs/senate/ + secțiunea din SKILL.md, consilium standard rămâne 100% funcțional.

§8 — Cross-questions (extensie viitoare — NU în MVP)

Status: documentată, neimplementată. MVP-ul curent e single-pass parallel. Secțiunea descrie vision-ul — pentru când acumulezi date care arată că merită cost-ul.

Cross-questions transformă Senatul din 7 voci independente într-un grup deliberativ care își poate clarifica pozițiile reciproc. În deliberarea istorică RUND2, 3 senatori și-au schimbat poziții post-cross-questions (Musk, Wittgenstein, Confucius) — semnal că dinamica funcționează.

§8.1 Cele 5 Legi ale Senatului

#LegeEsență
1Răspuns obligatoriuFiecare răspunde pe fiecare punct — direct, prin întrebare, sau prin reformulare. Interzis „nu am opinie".
2Cross-questions limitateMaximum 3 cross-questions per senator per punct. Previne spam și ramificare infinită.
3Blocaj → vot majoritar 5Dacă 2 senatori intră în impas, ceilalți 5 votează. 7 impar → niciodată 50/50.
4Sinteza doar la finalAggregator pe Senat rulează DUPĂ toate punctele (evită contagiunea de poziții).
5AuditabilitateToate runde + cross-questions + schimbări de poziție salvate în runs/senate/.

§8.2 Matricea cross-questions (cine cu cine)

Fiecare senator poate adresa întrebări fiecăruia altul. Matrice 7×7, diagonala goală. Marcate ★ perechile observate ca productive în RUND2 (au schimbat poziții).

WACSMuDN
Wittgenstein—••★•••
Aurelius•—•••★★
Confucius••—•★••
Socrate★••—•••
Musk••★•—•★
Dimon•★•••—★
Napoleon•★••★★—
★ a schimbat poziții în RUND2 (productivă empiric)
• pereche posibilă (toate sunt permise)
— self-cell (nu te poți întreba pe tine)

De ce sunt productive perechile ★: Wittgenstein↔Socrate (semantic vs asumpții — overlap pe „ce e nedeclarat"), Aurelius↔Dimon/Napoleon (risc abstract vs concret), Musk↔Napoleon (cost calitativ vs cantitativ), Musk↔Confucius (delete vs precedent).

§8.3 Flow runde (max 3 per senator per punct)

Runda 1
Inițial paralel

Toți 7 răspund simultan pe input identic.

  • Output structurat per senator
  • Voturi inițiale + concerns
  • Posibil cross-questions emise
→
Runda 2 (dacă există cross-Qs)
Dispatch focal

Doar senatorii întrebați răspund. Counter cross_q_used+1.

  • Input: întrebarea + contextul precedent
  • Output: răspuns + opțional nou cross-Q
  • Verifică cross_q_used < 3
→
Runda 3 (max)
Ultimul cross-Q

După 3 cross-Qs, STOP forțat per senator pe acel punct.

  • Răspuns final fix
  • Dacă încă blocaj cu altă voce → §8.4

▼
sinteză finală via senate_synth.py

§8.4 Blocaj resolution (Lege 3)

§8.5 Ce înseamnă pentru orchestrator când implementezi

Capability nouăSchimbare față de MVP
Dispatch multi-roundNu doar 1 ciclu paralel — 2-3 cicluri secvențiale; runde 2-3 doar pentru senatorii cu cross-questions
State trackingCounter cross_questions_used[senator][point] persistent în orchestrator
Routing cross-questionsParser pe output pentru „întrebare către <senator>" → dispatch focal
Blocaj detectionOrchestrator detectează 2 voturi opuse persistente după 3 runde → dispatch celor 5 senatori neutri
Aggregator schema schimbatvote_counts → vote_counts_per_round[] cu position_changes[] track-uite

Cost estimat: ~2-3× MVP-ul curent (per senatorii care intră în cross-questions; restul ies devreme).

§8.6 Indicator de calitate: schimbări de poziție

În RUND2 reală, 3 senatori și-au schimbat poziții post-cross-questions. Asta e semnalul că dinamica funcționează — nu schimbi opinia oarbă, schimbi după ce auzi un counter-argument concret. Un Senat care converge mereu fără schimbări de poziție e suspect (groupthink prin structură).

  }
]

§10 — Smoke test

Două nivele de verificare end-to-end:

1. Synth pe fixture

cat scripts/senate_synth_fixture.json | python -X utf8 scripts/senate_synth.py

Așteptare:

  • verdict: MODIFY
  • vote_counts: {GO: 3, MODIFY: 3, STOP: 1}
  • warnings: ["senator 'dimon' voted but omitted/empty 'stress_scenarios' — that axis of audit is silent"]

2. Suite completă (9 teste)

python -X utf8 scripts/test_senate_synth.py

Cazuri acoperite:

  • prompts_consistent — 7 fișiere markdown valide + format consistent (Rol/Specialitate/Output/Limite)
  • fixture_smoke — verdict așteptat din fixture
  • verdict_go_unanimous — 7/7 GO → GO
  • verdict_go_quorum — 5/7 GO + 2 MODIFY → GO
  • verdict_modify_default — 4 GO / 3 MODIFY → MODIFY
  • verdict_unreachable — 4 prezenți / 3 absenți → UNREACHABLE + warning
  • unrecognized_vote — vot invalid → contat MODIFY + warning structural
  • bundle_persisted — runs/senate/*.json scris cu keys obligatorii
  • collision_safe_write — fără overwrite silent; suffix _vN aplicat

Toate 9 trebuie să PASS înainte de orice commit pe senate_synth.py sau pe orice prompts/senators/*.md.
